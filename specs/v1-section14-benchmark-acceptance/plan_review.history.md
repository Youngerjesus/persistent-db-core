# Plan Verification Review: v1-section14-benchmark-acceptance

## Verdict
retry

The plan covers the main Section 14 scope, but it is not yet decision-complete for implementation. The next plan pass should resolve the gaps below before coding begins.

## Required Fixes

- [x] Decide how the benchmark will satisfy the contract schema `bench_items(id INTEGER PRIMARY KEY, group_key INTEGER, payload TEXT)` while the current SQL parser accepts `INT` but not `INTEGER`.
  - Evidence: `contracts.md` fixes `primary_key_type="INTEGER"` and the table schema with `INTEGER`; current `src/sql.rs` parses only `INT` in `parse_create_table`.
  - Required plan update: state whether implementation will add a narrowly tested `INTEGER` alias to the public SQL parser and docs, or keep SQL semantics unchanged and build the benchmark fixture through an internal storage/SQL helper that still emits the contracted schema evidence. Include the exact files/tests/docs affected by that choice.
  - Acceptance impact: without this decision, T4 can fail at DB setup or silently drift from the frozen fixture contract.

- [x] Resolve the evidence lifecycle for `commands.verify_bench_acceptance.status="pass"` inside `target/bench_acceptance/section14-benchmark-acceptance.json`.
  - Evidence: `contracts.md` requires the final evidence JSON to include both `commands.db_bench.*` and `commands.verify_bench_acceptance.*` pass fields, but `db bench` is the command that initially generates the evidence before `scripts/verify_bench_acceptance` has passed.
  - Required plan update: define which command writes or updates each `commands` entry, in what order, and how the script avoids either overclaiming verifier success before validation or invalidating the required `DB_BENCH` evidence path contract.
  - Acceptance impact: without an explicit ordering, implementation can produce evidence that either cannot be truthfully generated by `db bench` or lacks required command traceability after script verification.

- [x] Add an explicit negative regression test for the full-scan hard-fail path.
  - Evidence: the contract requires `FAIL-14-5` evidence from `scripts/verify_bench_acceptance` and a focused regression test. Current T3 checks that passing evidence says indexed paths were used, but does not require a test that fails when an eligible indexed equality/range path is reported or observed as a full scan.
  - Required plan update: add a task/subtask that injects or constructs a bad `index_use_evidence`/validator input, or otherwise forces the benchmark validator to see `observed_access_path=full_scan` for an eligible indexed query, and asserts the stable `check_id` failure and non-zero behavior.
  - Acceptance impact: without this, an implementation can pass by recording optimistic `hard_fail_checks` without proving the hard-fail rejection mechanism works.

## Non-Blocking Notes

- The existing `plan.md`, `design.md`, and `tasks.md` correctly identify the public `db bench` route, old 1k verifier replacement, required docs, repo-outside script invocation, and protected-area guardrails.
- The current repo already exposes `sql::plan_query_path_for_test`, so indexed-path proof is feasible, but the plan should still keep the negative hard-fail coverage explicit.
