# Fresh Repair Baseline

## Trigger Finding

`impl_brake_review.md` records `Fresh Repair Required: yes` with open findings
`IB-001`, `IB-002`, and `IB-003`.

## Repair Diagnosis

`IB-001`: The generated Section 14 evidence has the right shape but its latency,
file-size, index-use, and recovery values are synthetic. Primary root cause:
the benchmark writer constructed deterministic rows in memory and then hardcoded
metrics instead of exercising the persisted database workload. Secondary cause:
the evidence schema did not carry raw measurements that would make fabrication
obvious. Repair strategy: make `db bench` create persisted Section 14 databases
through the existing SQL path, run fixed indexed query sets against the database,
run explicit full-scan comparisons over the same deterministic fixture as the
contracted scan mode, measure medians from raw samples, and record actual DB/WAL
file sizes plus recovery reopen evidence.

`IB-002`: The verifier trusted precomputed pass fields and rewrote the evidence
to pass. Primary root cause: validation read already-derived fields without
independently recomputing medians, speedups, runtime cap, recovery bound, or
hard-fail rows from raw evidence. Repair strategy: add raw measurement arrays
and raw access/recovery facts to the JSON, then make `scripts/verify_bench_acceptance`
recompute formulas and hard-fail checks before finalizing the evidence.

`IB-003`: The Rust validator used brittle string matching over JSON text.
Primary root cause: no structured parser was used, so validation depended on
literal formatting and overbroad substring checks. Repair strategy: replace
validator logic with a small structured extractor for benchmark evidence fields
and index-use entries, sharing the same check semantics expected by the script.
