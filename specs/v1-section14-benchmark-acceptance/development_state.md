Repair findings addressed:
- `IB-001`: replaced synthetic hardcoded metrics with real persisted fixture creation and measured indexed/full-scan/recovery evidence.
- `IB-002`: verifier now recomputes acceptance formulas from `raw_measurements` and structured evidence before writing `result=pass`.
- `IB-003`: Rust validator now extracts structured index-use entries instead of relying on broad full-JSON substring matching.
- `IB-005`: evidence now records complete measured predicate proof coverage: 100 `secondary_equality` proofs and 50 `secondary_range` proofs, and both Rust validator and `scripts/verify_bench_acceptance` reject missing or partial coverage.
- `IB-006`: evidence now records structured workload provenance for fixed query keys, range windows, per-run counts, scan comparison mode, selectivity, sequential inserts, and recovery transaction count; `scripts/verify_bench_acceptance` independently validates those fields against Section 14 formulas.
- `IB-010`: benchmark-backed integration tests now serialize heavyweight command execution with an in-process lock, clean stale Section 14 test artifacts while holding that lock, tolerate prior panic lock poisoning, and snapshot command evidence before releasing the lock. Fixed query parsing is excluded from timed indexed batch execution so the benchmark measures the fixed workload execution path without parser noise.
- `IB-010` follow-up: internal workload/recovery databases are now process-isolated under `target/bench_acceptance/runs/section14-*-<pid>.pdb` and cleaned by the benchmark harness, while the public evidence path remains `target/bench_acceptance/section14-benchmark-acceptance.json`. This prevents stale or concurrent verifier attempts from corrupting the next benchmark run's internal DB state.
- `IB-012`: benchmark-invoking integration test binaries now share a cross-process test lock at `target/bench_acceptance/.section14.test.lock`, so `bench_acceptance` and `cli_contract` cannot clean or run benchmark artifacts concurrently under `scripts/verify`. Product `db bench` keeps its separate public evidence lock.
- `IB-015`: `scripts/verify_bench_acceptance` no longer bypasses the public `db bench` lock owner with `DB_BENCH_LOCK_HELD`; public `db bench` owns `.section14.lock` and writes lock owner pid metadata for stale-owner recovery. Benchmark-backed tests and the verifier script share `.section14.test.lock` only for canonical evidence cleanup/finalization serialization, and tests no longer delete shared product lock or run directories.
- `IB-016`: verifier finalization now writes the pass-updated JSON to a same-directory temporary file, flushes/fsyncs it, and atomically renames it over `target/bench_acceptance/section14-benchmark-acceptance.json`.
- `IB-014` realized as a full-verify flake: explicit full-scan comparison now scans full row payload bytes instead of only checking `group_key`, and benchmark evidence emits nine-decimal latency/speedup precision so strict formula recomputation does not fail due rounding drift on tiny indexed medians.
- `IB-017`: `verify_root_result()` and `verify_outside_result()` now take the in-process `BENCH_TEST_LOCK` before launching `scripts/verify_bench_acceptance`, while leaving the script-owned `.section14.test.lock` to the script itself. This prevents the `bench_acceptance` test binary from starting root and outside verifier scripts concurrently and self-timing out on the verifier lock.
- `IB-018`: verifier/test lock wait budgets now cover contaminated retry windows, and stale product lock pid probing suppresses `kill -0` stderr so stale lock cleanup cannot violate the public `db bench` empty-stderr success contract. Focused `bench_acceptance` was rerun from a state containing stale product lock/run artifacts and passed.
- `IB-019`: report-layer traceability now includes a measured Section 14 metric snapshot in `docs/performance_report.md`, and `docs/bug_diary.md` records the benchmark-facing storage append/replay cursor and WAL replay-scaling fixes with cause, fix, and regression status.
- `IB-022`: Rust writer-side Section 14 validation now scopes recovery proportionality to the `recovery` object, so `recovery_ms <= max(2000, recovery.wal_file_bytes / 4096)` cannot accidentally use `metrics.wal_file_bytes`. Regression coverage uses mismatched metrics/recovery WAL sizes and rejects the evidence with `recovery_proportionality`.
