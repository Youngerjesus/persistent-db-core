# Technical Design: v1-deterministic-crash-matrix

## Overview
The implementation should add a deterministic integration-level crash matrix around the current page-store WAL. The matrix owns evidence, not new database functionality. It should prove that rows interrupted before commit remain invisible, committed WAL rows replay exactly once, incomplete tails do not panic, and recovery interruption remains idempotent.

## Components
| Component | Responsibility |
|---|---|
| `tests/crash_matrix.rs` | Black-box CLI reopen assertions and fixture byte authoring |
| `tests/fixtures/crash_matrix/` | Required stable fixture manifest/README or seed descriptors for named cases, even when bytes are code-generated |
| `scripts/verify_crash_matrix` | Contract runner and markdown report writer |
| `target/crash_matrix/crash_matrix_report.md` | Generated evidence report |
| `docs/file_format.md` | Durable WAL compatibility note |
| optional private storage hook | Only if CM-005 cannot be expressed with fixtures alone |

## Fixture Encoding
Use deterministic helpers local to `tests/crash_matrix.rs`:
- `temp_db_path(case_id)`
- `wal_path(db_path)`
- `exec_sql(db_path, sql)`
- `seed_committed_one(db_path)`
- `sql_catalog_record(table, columns)`
- `sql_row_record(table, values)`
- `wal_frame(frame_id, record_count_before, state, payload_kind, payload)`
- `partial_wal_frame(...)`
- `corrupt_tail_bytes()`

The SQL record and WAL frame bytes should match current documented formats in `docs/sql_subset.md` and `docs/file_format.md`. If implementation instead exposes a repo-local test fixture helper, keep it private and avoid changing the library public API.

`tests/fixtures/crash_matrix/` must always be created. If binary fixture files are not useful, add a tracked manifest such as `tests/fixtures/crash_matrix/README.md` that lists:
- `seed_committed_one`
- CM-001..CM-006
- crash point labels
- evidence IDs
- expected visible rows
- whether the bytes are generated by `tests/crash_matrix.rs`

## Reopen Query
The spec examples mention `SELECT * FROM items ORDER BY id`, but current durable CLI contract does not include general `ORDER BY`. Use the approved equivalent "test harness's same CLI path" by creating `items (id INT PRIMARY KEY, name TEXT)` and running:

```text
db exec <db_path> "SELECT * FROM items;"
```

Primary-key table scans are documented as ascending primary-key order, so stdout remains deterministic:

```text
id|name
1|seed
```

or:

```text
id|name
1|seed
2|committed_wal
```

## Matrix Assertions
Each case assertion must include:
- `case_id`
- crash point label
- required evidence id
- reopen command string
- expected visible rows
- actual stdout rows
- stderr and exit status
- WAL/file-format assertion

Use assertion messages shaped like:

```text
CM-004 committed-wal-before-data-apply: expected idempotent rows after second reopen; stdout=...
```

## CM-003 Commit Marker Mapping
The current WAL format represents commit state in the frame header `state` byte. There is no separate commit marker after the WAL payload. CM-003 must model "WAL frame without commit marker" as a complete page-append frame whose state is `0x02` (`WAL_STATE_ROLLED_BACK`), not `0x01` (`WAL_STATE_COMMITTED`).

The test and report must record this mapping explicitly. Unknown state bytes are not an acceptable CM-003 fixture because current replay treats them as storage corruption, not as an accepted uncommitted frame.

## CM-005 Recovery Interruption Design
Preferred fixture-only approach:
1. Seed page file with catalog and row 1.
2. Write committed WAL frames for rows 2 and 3.
3. Manually apply row 2 to the page file while leaving both committed WAL frames in the sidecar.
4. Reopen through CLI.
5. Current replay should skip already-applied row 2 because durable record count is greater than frame 1's `record_count_before`, then apply row 3.

This models interruption after first recovery apply and before WAL cleanup/checkpoint because the data file contains the first recovered row while the WAL sidecar still contains the full replay prefix.

Fallback hook approach:
- Add a private/test-only interruption point after first committed recovery apply.
- Invoke it only from tests.
- Ensure the hook leaves no user-facing CLI option and no runtime behavior change for normal execution.

## CM-006 Corrupt Tail Design
CM-006 is fixed to the no-CLI-change success path. The fixture must contain:
1. `seed_committed_one` in the page file.
2. A complete committed WAL frame for `(2, 'committed_before_tail')`.
3. A deterministic trailing fragment that replay classifies as incomplete, such as fewer than 36 header bytes or a header/payload prefix whose declared payload length exceeds available bytes.

Expected result:
- reopen exits `0`,
- stderr is empty,
- stdout contains exactly rows 1 and 2 in deterministic order,
- repeated replay remains idempotent if the test repeats reopen,
- the report states the incomplete/invalid-length tail was ignored/truncated per `docs/file_format.md`.

Do not use a complete invalid-magic, unsupported-version, checksum-invalid, or unknown-state frame for CM-006. Those remain existing storage-corruption behavior and are not this task's success path.

## Report Generation
`scripts/verify_crash_matrix` should:
1. Resolve `repo_root` from the script path and `cd` there.
2. Create `target/crash_matrix`.
3. Run `cargo test --test crash_matrix`.
4. Collect observed per-case results from the executed matrix run. Acceptable designs are:
   - `tests/crash_matrix.rs` writes a machine-readable case result file under `target/crash_matrix/` during the test run, and the script renders markdown from it.
   - the script invokes the same crash matrix harness in a report mode that executes each case and records observed rows/status directly.
5. Write `target/crash_matrix/crash_matrix_report.md` from those observed case results.
6. Validate the report contains CM-001..CM-006 and `crash-matrix-case-CM-001`..`crash-matrix-case-CM-006`.

The report must not be a purely static success template. Each row's actual visible rows and command exit status must come from the executed case result.

## Documentation Design
Update `docs/file_format.md` only for WAL crash compatibility details that are not already durable:
- no WAL sidecar or empty sidecar opens normally,
- incomplete trailing WAL header/payload is ignored and truncated,
- committed prefix before incomplete tail is replayed,
- retained complete frames are idempotent,
- complete corrupt frames remain storage corruption unless implementation changes that behavior.

Do not update `docs/cli_contract.md` unless implementation changes user-facing output, stderr, or exit codes.
