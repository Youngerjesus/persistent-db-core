# Research: v1-deterministic-crash-matrix

## Decision 1: Keep Crash Matrix As Black-Box Integration Coverage With Deterministic WAL Fixtures
Use `tests/crash_matrix.rs` as the owning test surface. It should invoke the built `db` binary through `env!("CARGO_BIN_EXE_db")` for reopen checks and should author deterministic WAL sidecar bytes only where no public CLI can create the crash boundary.

Rationale:
- The contract requires CLI-equivalent reopen behavior and deterministic `SELECT` output.
- V1 has no public transaction or crash command, so fixture-authored WAL bytes are the narrowest way to model pre-commit, partial, recovery-interrupted, and corrupt-tail states.
- Existing `tests/wal_recovery.rs` already demonstrates SQL record and WAL frame fixture helpers, so the crash matrix can reuse the local pattern without broadening dependencies.

Rejected alternative:
- Adding public crash-injection CLI flags would change the user-facing CLI surface and would require `docs/cli_contract.md` changes outside the approved need.

## Decision 2: Add Minimal Test-Only/Internal Crash Injection Hooks Only If Fixtures Cannot Express A Boundary
Prefer deterministic fixtures for CM-001, CM-002, CM-003, CM-004, and CM-006. For CM-005, implementation may need a narrow internal hook around recovery apply/cleanup if fixture composition alone cannot prove "interrupted after first apply". The hook must stay non-public and must not change CLI output.

Rationale:
- The spec allows "necessary minimal crash injection hook".
- Recovery interruption is the only case that may require observing a mid-replay boundary.
- Keeping hooks non-public preserves the documented CLI contract.

Implementation constraint:
- If a hook is added, guard it behind test-only code or a private API used only by integration tests through deterministic fixture setup. Do not add network services, daemons, or public CLI switches.

## Decision 3: Resolve CM-006 To The Existing No-CLI-Change Success Path
`docs/file_format.md` currently states incomplete trailing frames are ignored and truncated, while complete frames with invalid magic, unsupported version, checksum mismatch, or unknown non-rollback state fail open as storage corruption. For this task, CM-006 must use a deterministic incomplete or invalid-length trailing WAL fragment after a committed prefix so reopen succeeds and the visible rows are exactly `[(1, 'seed'), (2, 'committed_before_tail')]`.

Planning consequence:
- Implement CM-006 using trailing bytes that are incomplete from replay's perspective, such as a short header or a header that declares a longer payload than the bytes present.
- Do not use a complete checksum-invalid frame or complete invalid-magic frame for CM-006. Those remain existing storage-corruption behavior and are out of scope for this crash matrix unless a future spec changes the contract.

## Decision 4: Map CM-003 "Commit Marker" To The Current WAL State Byte
The current WAL format has no separate post-payload commit marker. Commit state is represented by the frame `state` byte in the WAL header: `0x01` committed and `0x02` rolled back. CM-003's "WAL frame without commit marker" should therefore be implemented as a complete current-format page-append frame for row 2 with state `0x02` rolled back, which replay accepts and skips.

Planning consequence:
- The test and final report must explicitly document this mapping: "commit marker absent" means "not `WAL_STATE_COMMITTED`; represented by `WAL_STATE_ROLLED_BACK` in the current V1 WAL format".
- Do not invent an unknown state byte for CM-003, because the current replay treats unknown non-rollback state as storage corruption rather than as a skipped uncommitted frame.

## Decision 5: Report Generation Must Be Derived From The Executed Matrix Run
Create `scripts/verify_crash_matrix` as the deterministic matrix runner. It should run the crash matrix harness and write `target/crash_matrix/crash_matrix_report.md` with one row per executed case. The report must be populated from observed per-case results, not from a purely static success template.

Rationale:
- The contract requires both test command evidence and a durable local report path.
- The report contract requires actual visible rows and command exit status for each case, so `scripts/verify_crash_matrix` must either consume per-case result records emitted by `tests/crash_matrix.rs` or run the same harness itself and record the observed stdout rows/status.

## Decision 6: `tests/fixtures/crash_matrix/` Is Unconditional Evidence
The implementation must create `tests/fixtures/crash_matrix/` even if WAL bytes are generated by Rust helpers. At minimum, add a tracked `README.md` or manifest that names `seed_committed_one`, CM-001..CM-006, crash point labels, expected rows, and evidence IDs.

Rationale:
- `contracts.md` lists the directory as a required after-implementation evidence path.
- A manifest keeps fixture identity stable while avoiding brittle binary fixture files when code-generated WAL bytes are clearer.

## Decision 7: No New Dependency Is Needed
Use standard library temp directories, file IO, `std::process::Command`, and simple byte encoders. Do not add crates for fixtures, reports, or checksum.

Rationale:
- AGENTS.md prefers standard library implementations for V1 unless a spec explicitly calls for a crate.
- Existing tests already implement deterministic helpers with std only.

## Open Risks
- `ORDER BY` appears in the spec reopen command examples, while durable docs currently exclude general `ORDER BY`. Tests should use an equivalent deterministic CLI path that is valid for the current SQL subset, likely primary-key table scan order or `SELECT * FROM items;`.
- Private WAL constants/functions are duplicated in existing tests. Crash matrix tests may duplicate the fixture encoder unless implementation introduces a test-support module without expanding public API.
