It does not depend on generated launch evidence, screenshots/browser artifacts, exported reports, evaluator output, redaction proof, or artifact-producing harnesses.
- Provenance Contract: not applicable for this phase. Current-run command evidence still must be regenerated by implementation/verification phases and must not reuse historical command output as current proof.

## Scenario Expansion Lens
- Happy path: create `INT PRIMARY KEY` table, insert out-of-order keys, exact lookup, ordered `SELECT *`, reopen/rebuild.
- Invalid input: `TEXT PRIMARY KEY`, unsupported non-PK `WHERE`, unsupported range predicate, unsupported `ORDER BY`, malformed SQL remains outside this slice.
- Partial state: empty PK table scan and missing key lookup return header only.
- Duplicate/already-done: duplicate PK insert fails before append; duplicate persisted PK row discovered during rebuild fails as invalid SQL storage record.
- Dependency failure: corrupt SQL logical records continue through the existing invalid SQL storage record path.
- Permission/trust-boundary: no new trust boundary, network, secrets, browser, or external service is introduced; filesystem page-store errors stay under baseline `scripts/verify` and existing storage tests.
- Retry/re-entry: after a failed command, prior committed statements remain persisted and later statements are skipped; PK duplicate tests assert no duplicate row is appended.
- Compatibility: existing row-only catalog/table files without PK metadata load as no-primary-key tables and keep insert-order scans.
